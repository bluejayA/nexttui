NEXT 도입 이후 운영 작업 예상 변화

다음은 Classic 와 NEXT의 운영 작업 프로세스의 차이에 대한 이야기를 간단하게 설명합니다. 

[AS-IS] Classic 인프라 관리 시스템에서의 운영 체계

Classic 인프라 관리 구조에서는 OpenStack이 자원 상태와 데이터의 단일 기준(Source of Truth) 로 동작하였습니다.

이 구조에서 OpenStack CLI를 통한 작업은 곧 기준 데이터를 직접 수정하는 행위였으며,
포탈 DB, NCS, NC 등의 나머지 시스템은 OpenStack의 상태를 기준으로 이를 정렬하는 방식으로 운영되었습니다.

따라서 시스템 간 정합성은 사후에 OpenStack 상태를 기준으로 맞추는 문제였고,
이는 각 시스템 별 운영자의 숙련도와 책임 하에 관리 가능한 영역으로 인식되었습니다.

[TO-BE] NEXT 인프라 관리 시스템의 구조 변화

NEXT 인프라 관리 시스템에서는 OpenStack이 더 이상 단독 기준 시스템으로 동작하지 않습니다.

OpenStack과 함께

서비스 DB

VPC 제어 계층

정책 및 승인 체계

가 동시에 기준 시스템의 일부로 작동하는 구조로 전환되었습니다.

이로 인해 변경 작업은
Classic과 같이 특정 시스템에서 먼저 수행한 뒤 나머지를 맞추는 방식이 아니라,
사전에 여러 기준 간 정합성을 보장해야 하는 구조가 되었습니다.

이러한 구조 변화로 인해
Classic에서 가능했던 OpenStack 직접 접근 방식은
NEXT 환경에서는 더 이상 동일하게 적용하기 어려워지게 되었습니다.

운영 행위는 더 이상 특정 시스템의 단순 변경 작업이 아니라,
여러 기준 시스템을 동시에 고려한 운영 판단 단위로 다뤄져야 하며,
이에 대한 명확한 기준과 체계가 요구되는 상황입니다.

이러한 변화는 기존 운영 도구와 접근 방식만으로는 대응하기 어렵다는 점을 명확히 보여주며,
이에 따라 NEXT 환경에 적합한 운영 시스템 전략의 재정의가 필요해졌습니다.

NEXT 운영 시스템 전략

NEXT 운영 시스템 전략 Workshop 회의: https://ktcloud.atlassian.net/wiki/spaces/TECHKCDC/pages/513474988 

이에 따라 kt cloud의 NEXT Cloud 개발에 따른 운영 시스템 전반의 재정비와 고도화에 대한 WorkShop을 작년 7월에 진행하였고, 
해당 Workshop에서 논의된 NEXT 운영 시스템의 전체 그림은 다음과 같습니다.

이번 NEXT Admin CLI의 기획 범위는 위 그림의 OSS 영역에 있는 NEXT Admin Console(붉은 박스)을 제공하기에 앞서, 
우선적으로 상기 체계의 운영 편의성을 빠르게 확보하기 위한 CLI 형태의 서비스를 제공하는 것을 주요 기능으로 선정하고 개발할 예정입니다.

다만 NEXT Admin Console의 제공은 특성 상 NEXT Admin API(푸른 박스)에 대한 기획 및 기능 개발과 궤를 같이 하기 때문에
Admin CLI 기획 및 개발 또한 Admin API와 연계되어 하나의 시스템 형태로 기획 및 개발이 필요합니다.

그래서 기획안에서 다루게 되는 Scope은 NEXT Admin CLI 및 API 까지를 아우를 예정입니다.

실제 운영 실무자 관점에서는 어떤 어려움 혹은 개선을 바라고 있는지,
그리고 현업에서는 어떻게 이를 인식하고 있는지를 확인하기 위해 일부 실무 리더들을 대상으로 포인트 인터뷰를 진행하였습니다.

인터뷰

계획/목표

우선 1월에는 1차적으로 인터뷰 대상 영역을 OpenStack과 연관성이 높은 서비스 개발 영역, 시스템 설계 영역, 시스템 운영 영역으로 분리하여 
위와 같은 변화의 방향성을 공유하고, 각 영역별로 관련 작년, 올해 테크리더 혹은 운영 파트 리더를 담당하고 있는 대상자들에게 Admin CLI에 대한 생각을 공유하고 의견을 받는 것을 목표로 1차 진행하였습니다.

2월부터는 IaaS 관련 부서 외 데이타 플랫폼 / Infra Ops 등 추가 연관 영역을 점차 넓혀 인터뷰를 진행하고 
실무자들과의 지속적인 커뮤니케이션을 통해, 실제 설계 및 개발에 반영될 수 있도록 관련 커뮤니케이션을 지속해 나갈 예정입니다.

이번 인터뷰 주요 내용

우선 Admin CLI 기반 시스템을 도입함에 있어 위 영역에서 가장 크게 영향을 받을 만한 팀들을 간추려 
주요 필요 기능 사항에 대한 요구사항을 수합하는 것을 첫번째 목적으로 진행하였습니다. 

대상 팀으로 추려진 플랫폼개발팀, Virtual네트워크팀, Core플랫폼팀, SW운영팀, HW운영팀 대상으로 
실무리더급 담당자들에게 Admin CLI에 대한 요구사항 혹은 영향도에 대한 내용을 질의하였고,
관련 대화 결과에 대한 정리를 해보자면 크게 3가지로 다음과 같습니다.

Key Sentence 1) 빈번하게 반복되는 운영 작업을 안정적으로 지원하는 것이 중요

우선 인터뷰를 통해 운영자가 관리자 시스템을 활용하는 상황은 전체 기능 범위 대비 일부 특정 시나리오에 집중되어 있음을 확인할 수 있었습니다.

운영 상황은 대체로

자원 정리·최적화와 같은 정기 운영 작업

장애, 비정상 동작 등으로 인한 수작업 조치

고객 요청에 따른 예외 처리 작업

의 세 가지 유형으로 구분되었으며,
이 중 고객 요청에 따른 예외 처리나 장애 대응은 상황 의존적이며 발생 빈도가 상대적으로 낮은 편입니다.

반면, 자원 정리·최적화와 같은 정기 운영 작업은 운영 과정에서 반복적으로 수행되고 있으며,
운영 효율과 안정성 측면에서 기능 제공 우선순위가 가장 높은 영역으로 판단됩니다.

이는 관리자 시스템이 모든 가능 시나리오를 포괄하기보다는,
빈번하게 반복되는 운영 작업을 안정적으로 지원하는 방향으로 우선 설계될 필요가 있음을 시사합니다.

Key Sentence 2) 조회를 제외한 생성/변경/삭제의 운영 작업은 서비스 시스템과의 정합성 이슈를 동반할 가능성이 높거나 필연적임

조회성 업무를 제외한 생성·삭제·변경과 같은 운영 작업이 발생할 경우
OpenStack 단독 조치만으로는 작업이 완료되지 않으며,
서비스 GW 및 서비스별 관리 데이터와의 정합성을 유지하기 위한 추가적인 프로세스가 필요하다는 의견이 다수 제기되었습니다.

특히 서비스 개발 부서별로

제공 중인 기능 범위

관리 중인 데이터 구조

OpenStack과의 연계 방식

이 상이하다 보니,
동일한 유형의 운영 작업이라 하더라도 서비스별로 정합성 확보 방식과 영향도가 크게 다르게 나타남을 확인할 수 있었습니다.

일부 서비스의 경우
단순한 상태 정보 반영이나 데이터 업데이트로 처리가 가능한 반면,
다른 서비스에서는 구조적으로 서비스 레벨의 제어 없이는
운영 작업 수행이 어렵거나 불가능한 경우도 존재하는 것으로 파악되었습니다.

이는 운영 작업을 일괄적인 방식으로 처리하기 어렵고,
운영 행위에 앞서 서비스별 영향도와 정합성 수준을 고려한 판단이 선행되어야 함을 시사합니다.

Key Sentence 3) 운영자의 주요 부담은 변경 행위가 아닌 사전 판단에 있음

운영팀이 공통적으로 언급한 어려움 중 내용을 분석해보면,
자원을 실제로 변경하는 행위 자체보다 영향도 등의 파악을 주요 목적으로 한 필요한 정보를 사전에 확보하기 어렵다는 점에 집중되어 있었습니다.

특히,

VM과 연결된 Block/NAS 자원의 관계를 한 화면에서 확인하기 어려움

동일 자원의 식별을 위해 OpenStack, Manila, 스토리지 시스템을
각각 확인해야 하는 구조

단순 현황 파악에도 특정 담당자의 개입이 필요한 상황

이 Classic에서의 불편함으로 대두 되었습니다.

이는 운영자가 필요한 정보가 존재하지 않아서가 아니라,
정보가 시스템별로 분산되어 있어 운영 판단에 필요한 맥락을 한 번에 파악하기 어렵기 때문으로 해석됩니다.

결과적으로 운영자의 부담은
변경 작업의 난이도보다는 사전 판단을 위한 정보 접근 구조에서 기인하고 있음을 시사합니다.

조사 사례 

Naver Cloud

Naver Cloud는 운영자 관점에서 사용할 수 있는
통합 관리자 포탈 형태의 서비스를 제공하는 구조를 채택하고 있습니다.

이 관리자 포탈은 단일 시스템이 모든 로직을 담당하는 형태가 아니라,
다음과 같은 역할 분담 구조를 가집니다.

CMDB 기반 자원·고객·서비스 정보 관리

자원 조회, Quota 변경, 운영 조치 등의 관리 기능을 제공하는 Admin API

관리자 포탈은 위 기능들을 호출·조합하는 UI 계층

운영자가 사용하는 기능은 포탈에 통합되어 있으나,
실제 운영 기능의 구현과 확장은 Admin API 및 CMDB 계층에서 이루어집니다.

즉, Naver Cloud의 운영 구조에서는
Admin API가 운영 기능의 실질적인 제공 주체이며,
포탈과 자동화 도구는 이를 소비하는 형태로 구성됩니다.

이는 Admin API/CLI 기반 운영 기능을
서비스 형태로 제공하는 국내 사례로 볼 수 있습니다.

AWS

AWS는 Admin API 및 CLI를 운영자 전용 도구로 분리해 정의하기보다는,
운영자가 수행해야 하는 관리 기능을 목적별 관리 서비스 형태로 제공합니다.

AWS CLI는 존재하지만,
운영 기능은 CLI 자체에 모이지 않고 다음과 같은 관리 성격의 서비스로 분산되어 있다.

계정·조직 관리 서비스
AWS Organizations, IAM을 통해 계정 구조, 권한, 운영 범위를 관리

운영 상태 조회 및 영향도 확인 서비스
AWS Health / Personal Health Dashboard를 통해
고객 계정 단위의 장애 영향, 계획 작업, 리스크 상태를 조회

정기 점검 및 최적화 서비스
Trusted Advisor를 통해 비용, 성능, 보안, 안정성 항목에 대한
정기 점검 결과를 제공

운영 표준 적용 및 계정 관리 서비스
Control Tower를 통해 계정 생성, 보안 기준, 로깅 설정을 표준화

이러한 서비스들은 내부적으로 Admin API를 기반으로 동작하며,
운영자는 콘솔, CLI, 자동화 도구를 통해 해당 API를 호출하는 구조를 가집니다.

즉, AWS에서 Admin API/CLI는
개별 리소스를 직접 조작하기 위한 도구라기보다,
운영자가 수행해야 하는 관리 행위를 서비스 단위로 제공하기 위한 기반 인터페이스로 사용됩니다.

CSP 공통 특징

위 사례들을 종합하면 다음과 같은 공통점을 도출할 수 있습니다.

CMDB 및 데이터 플랫폼 등과 연계되어 운영자에게 인사이트를 제공하는 기능을 제공

CLI 및 콘솔은 Admin API를 호출하는 접근 수단이며 Admin API가 이러한 관리 서비스를 구성하는 공통 기반 인터페이스를 제공하는 구조 

계정/조직 관리, 운영 상태 조회, 정기 점검, 정책 적용과 같이 운영 목적별로 분리된 관리 서비스 형태로 관리 및 제어

운영자가 경험하는 기능은 개별 리소스 조작이 아니라 조회, 매핑, 정기 운영 작업, 관리 단위 조치 중심으로 구성됩니다.

운영자가 수행하는 모든 관리 행위에 대해 권한 기반(RBAC) 접근 제어 및 감사가 가능한 CLI 및 API 구조를 제공한다.

서비스 표준을 유지하기 위해 Golden Image 생성 및 인스턴스 타입(Flavor)등의 고객에게 제공되는 정책 관리 기능을 제공한다.

고객 콘솔에 노출되지 않는 백그라운드 운영 작업(Fault 조치, 상태 보정, 내부 처리 등) 을 수행할 수 있는 관리 기능을 제공한다.

인프라 운영을 위해 Hypervisor 단위의 VM Migration(수동/계획 기반) 등의 인프라 작업을 수행할 수 있는 관리 기능을 제공한다.

운영자 역할에 따라 관리 기능 접근 범위를 제한하고 제어할 수 있는 권한 관리(RBAC) 기능을 제공한다.

해당 기능들을 구현하기 위해 크게 두가지 영역으로 분리하여 기획하는 것을 목표로 합니다.

영역 2. 전체 제공 기능 범위 정의 및 해당 기능을 위해 필요한 Admin API 분류

영역1. 아키텍쳐 설계 방향

즉, 운영자가 실제로 접근하는 방식에서 출발해, CLI/API의 책임 분리와 통제 지점을 정의한 뒤, 마지막으로 OpenStack 및 서비스GW 확장 까지 고려 합니다.

Admin CLI 접근 경로 

ktcloud 운영 환경에서는 단말로 VDI를 사용하여 운영 단말의 보안 수준을 중앙에서 통제 하고, 관리망 접근에 대한 일관된 정책 적용을 가능하게 한다.
이에 따라 운영자가 관리망에 접근할 수 있는 방식은 제한적이며, 해당 제약 조건을 전제로 Admin CLI 제공 방안을 검토한다.

1안. 가상 콘솔 기반 접근 방식

(Tag Agent RDP → 가상 콘솔에서 Admin CLI 실행)

개요

운영자는 VDI에서 Tag Agent가 제공하는 RDP 기능을 통해
관리망 내부의 가상 콘솔(운영 Tool Host) 에 접속한다.
Admin CLI는 해당 가상 콘솔에 설치되어 있으며, 관리망 내부에서 실행된다.

구성 개념

운영자
 └─ VDI
      └─ Tag Agent (RDP)
           └─ 가상 콘솔 (Windows / MGMT)
                └─ Admin CLI
                     └─ Internal API GW
                          └─ Service Layer


특징

VDI는 단말(UI) 역할만 수행

관리망 접근은 Tag Agent가 제공하는 승인된 RDP 채널을 통해서만 가능

Admin CLI는 관리망 내부에서 실행

현행 보안/망 구조를 그대로 유지

장점

현행 보안 정책과 100% 정합

Tag Agent 개발/확장 불필요 (기존 기능 활용)

향후 Admin Console(웹 기반 운영 콘솔) 로 자연스럽게 확장 가능

단점 / 부담

홉 증가로 인해 1안 대비 UX/응답성은 다소 저하

추후 Admin Console 등 제공 시, 느린 화면 응답 속도로 사용성에 문제될 수 있음

이미지 업로드 등 대용량 파일 작업에 대한 접근성이 어려워 질 수 있음

2안. VDI 직접 접근 방식

(VDI → Internal API Gateway 직접 통신)

개요

운영자는 사내 VDI(Windows) 환경에서 Admin CLI를 직접 실행하며,
VDI에서 Internal API Gateway로 HTTPS 통신이 허용되는 것을 전제로 한다.
Admin CLI는 별도의 중간 호스트 없이 관리망 API에 직접 접근한다.

구성 개념

운영자
 └─ VDI (Windows)
      └─ Admin CLI
           └─ Internal API GW
                └─ Service Layer


특징

Admin CLI는 VDI에 설치된 로컬 실행 도구

관리망 접근은 Internal API GW 단일 엔드포인트로 제한

Tag Agent는 관여하지 않음 (장비 접속과 분리)

장점

홉이 비교적 적음 → 응답 속도/사용자 경험(UX) 우수

운영 환경이 단순하고 직관적

자동화/스크립트 연계에 유리

단점 / 제약

VDI에서 관리망으로의 직접 HTTPS 접근이 허용되어야 함

네트워크/보안 정책 변경이 선행되지 않으면 적용 불가 → 보안부서와 검토를 통해 해당 형상 제공 가능 여부 확인

Admin CLI / Admin API 역할 범위 및 호출 책임

운영자 대상 Admin CLI는 단순 API 호출 구조가 아니라 
사전 점검 → 실행 → 검증 → 기록을 포함한 “운영 행위”를 안전하게 수행하는 것이 목적으로
이에 따라 개발 생산성, 정보 정합성, 보안·감사, 장기 기술부채를 함께 고려한 아키텍처 선택이 필요하여,
아래 2가지 개발 방안을 생각해 볼 수 있다.

정합성 유지를 위해 필요한 구성

영역2. Admin API 제공 기능 범위

Admin API 가 서비스를 제공해야 되는 대상과 영역이 Cloud 서비스 및 시스템의 대부분이므로 Scope을 분리하여
서비스 제공 정책을 계획하고 수립한다.

1차 Scope – IaaS 기반 운영 통제 (본 기획의 핵심 범위) 

1차 Scope는
NEXT Admin CLI / API가 직접적인 운영 통제 책임을 가지는 영역으로,
본 기획의 중심이 되는 범위이다.

OpenStack 기반 Cloud Platform
(Nova, Cinder, Neutron, Glance, Keystone)

IaaS 기반 Service Platform
(Server / Storage / Network 서비스 GW, VPC 제어 계층)

운영 통제 기능

프로젝트/고객 단위 Quota 관리

Flavor / Image 정책 관리

Hypervisor 단위 운영 작업

관리자 권한 관리(RBAC) 및 감사

본 현재 시점의 해당 문서에서 정의하는 주요 기능 및 아키텍처 검토는 이 영역이 주요 범위이다.

2차 Scope – 운영 판단 지원을 위한 연계 영역
